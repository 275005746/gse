# Capability Execution Matrix

Use this before opening or continuing a GSE core capability slice.

This matrix turns `.gse/goal-map.md` from a status ledger into an execution router. A core capability must map to GSE behavior, implementation routes, the current gap, the next smallest slice, and focused verification before implementation begins.

## Operating Rule

For GSE self-development:

1. Find the capability row before implementation.
2. If no row exists, add or update the row first.
3. Use the implementation route list as the minimum routing check.
4. Keep optional host-native or marketplace claims as `external-required` or `not-claimed` until real evidence exists.
5. Record evidence with an explicit `evidenceLevel` and `requiredEvidenceLevel`.

This matrix is not a replacement for `.gse/goal-map.md`; it is the pre-work router that keeps goals tied to implementation discipline.

## Keep It Lean

- Prefer one row per real capability family, not one row per doc variant or audit synonym.
- Merge rows when the same verification path, evidence level, and claim boundary apply.
- If a row only restates another row with a different filename, delete or fold it into the stronger row.
- Keep the matrix focused on core execution and claim boundaries, not on describing the entire GSE ecosystem.

## GSE Control Patterns

- Keep `SKILL.md` concise and route detailed behavior through `references/` and executable scripts.
- Define focused verification before changing core workflow behavior.
- Prefer root-cause repair for bug, failure, and drift slices.
- Keep claim boundaries and command routing machine-auditable.

## Minimum Row Requirements

Every row must include:

- a named capability,
- a trigger or "when to use" condition,
- a GSE behavior,
- implementation routes,
- current implementation status,
- known gap,
- next smallest core slice,
- required evidence level,
- focused verification command or check,
- claim boundary.

Rows may point to optional host tools, but optional tools must include a fallback or an explicit external/unknown boundary.

## Close Rule

A GSE self-development slice that changes core workflow behavior should not close unless either:

- the changed capability already has a current row in this matrix, or
- the slice adds/updates the row as part of the change.

Run:

```text
node scripts/audit-capability-execution-matrix.mjs --root . --json
```

before claiming a matrix-covered GSE capability is complete.
